## 0. 결론 아키텍처

```
[GitHub Public 리포]
  └ Actions cron */30 (무료·무제한, 수분~수십분 지연 감수)
      ├ 1. 사실 수집  : DART 공시 API + 시세 API + 공식 RSS → facts/*.json (출처 URL 필수)
      ├ 2. 분석 생성  : LLM에 facts JSON"만" 주입 → 마크다운 초안
      ├ 3. 기계 검수  : 초안의 모든 수치를 정규식 추출 → facts JSON과 대조 → 불일치 시 발행 차단
      └ 4. 커밋 → wrangler deploy → Cloudflare Workers Static Assets (요청 무제한 무료)
```

## 2. 사실성(100%) 확보 설계 — 검증된 원칙

**전제**: "100% 사실"이 보장되는 범위는 **데이터**(공시·시세·지표)다. **분석(해석)은 정의상 의견**이므로, 사실과 해석을 분리 표기하는 것이 유일하게 정직한 설계다.

검증된 근거:
- AP통신 실적기사 자동화(2014~): LLM이 아니라 **구조화 데이터 + 템플릿(NLG)** — 데이터에 있는 숫자만 문장에 들어가므로 구조적으로 환각 불가. 분기 300건→3,000건+, 사람보다 오류가 적었다고 보고. 국내 파이낸셜뉴스 fnRASSI·연합뉴스 사커봇도 동일 방식
- Anthropic 공식 환각 억제 기법: ①"모른다" 허용 ②원문 인용 추출 후 인용에만 근거해 작성 ③주장마다 뒷받침 인용 확인, 없으면 철회
- 실무 표준 4단 방어: 스키마 검증 → 주장 분해 → 별도 검증(생성 모델과 다른 규칙/모델)으로 원본 대조 → 불일치 시 재시도/차단

**적용 (collect_and_publish.py에 구현된 구조)**:
1. 수치·고유명사는 LLM이 생성하지 않는다 — 시세 스트립·공시 목록·표는 **템플릿에 데이터 슬롯 채우기**(AP 방식)
2. 분석 문단만 LLM — 입력은 facts JSON만, "JSON에 없는 숫자 생성 금지 + 모르면 확인 필요 표기" 프롬프트
3. 발행 전 기계 검수: 초안에서 숫자를 정규식으로 전부 추출해 facts JSON과 대조, 하나라도 불일치하면 그 회차 발행 차단(결번이 오보보다 낫다)
4. 해석 문장은 "~로 해석할 수 있다/~일 가능성" 어미로 사실 문장과 구분

## 3. 구글 SEO 정책 리스크 (공식 문서 확인)

- 구글 공식 원칙: **"제작 방식이 아니라 품질"** (2025-12-10 갱신 문서에서도 유지)
- 위반 기준(scaled content abuse): "순위 조작이 주목적 + 사용자 가치 없음". 데이터 피드 단순 변환 대량 발행이 정확히 위반 예시 → **30분 발행 자체는 위반이 아니지만, 부가가치(수혜/피해 구분, 반대 시나리오, 체크포인트) 없는 회차는 리스크**
- Quality Rater Guidelines: 자동화가 실질적으로 쓰였다면 **방문자에게 자명하게 고지**할 것 → 글 하단에 "이 글의 데이터는 공시·시세 API에서 자동 수집되었으며 분석은 편집 기준에 따라 생성·검수되었습니다" 고지 블록 권장
- YMYL(금융): 저자 식별·About·출처 인용·수정일이 사실상 필수 신호

## 4. 한국 규제 (금융위 유권해석 확인)

- **무료 공개 + 광고수익 모델 = 유사투자자문업 신고 대상 아닐 가능성 높음** (금융위 법령해석: 누구나 무료로 같은 정보를 보면 "직접적 대가"가 아님)
- **유료 멤버십/구독을 붙이는 순간 신고 대상** + 신고해도 1:1 개별 조언은 금지 → 수익화 4단계(유료 구독)로 갈 때 반드시 재검토
- 면책 문구는 법적 의무는 아니나 리스크 완화 관행 — 전 글 하단 유지

## 6. 설치 순서 (요약)

1. GitHub에 public 리포 생성 → 이 폴더 전체 push
2. `automation/publish.yml`을 `.github/workflows/publish.yml`로 이동
3. 리포 Settings → Secrets에 `DART_API_KEY`, `DATA_GO_KR_KEY`, `LLM_API_KEY` 등록
4. Cloudflare 계정에서 Workers 프로젝트 생성 → `wrangler deploy` 단계를 publish.yml에 추가
5. superstock.blog 도메인을 Workers 커스텀 도메인으로 연결
